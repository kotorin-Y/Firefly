## 前两段工作经历

第一份工作在互联网大厂，做本地生活商业化运营（BD）。当时需要跟进本地商户合作，参与节点活动，配合广告投放和社区内容运营。工作很杂，也很贴近业务现场：商户在意投入有没有回报，平台关心流量和交易，用户则只会为真正感兴趣的内容停下来。

我参与过大众点评必吃榜等活动，日常会看消耗、转化、社区活跃度和 GMV。活动期间，目标客户群社区周活跃度提升 40,000+，下单量 / 访问量转化率提升 13%，落地成本减少 20%；负责范围内核心商圈 GMV 月均同比提升 17%，平台市占率提高 8%。除了追结果，我也开始整理看板和 SOP，因为反复踩同一个坑实在没有必要。

后来去了教培大厂做教育顾问。工作对象从商户变成了家长和学生，沟通方式也完全不同。家长迟迟不做决定，可能是课程不合适，也可能只是信息太多、顾虑没有被听见。我参与过线上试听课、小鹅通讲座和社群跟进，慢慢学会把需求听完整，而不是急着给答案。

这段经历让我重新理解了“转化”这个词。数据当然重要，但数据后面是一个个具体的人。一次活动结束后该怎么继续沟通，一段内容为什么没有打中用户，服务体验在哪个环节开始变差，这些细节往往比一句“提升转化率”更值得追。

## 我怎样使用 AI

“会写提示词”并不能说明太多。提示词只是开始，后面还有资料质量、上下文、工具调用、结果检查和失败处理。

我常用 ChatGPT、Codex、Claude 和 Gemini。做项目时，通常会先把目标和限制说清楚，再检索资料、核对来源、生成初稿、人工修改，最后通过测试或实际操作验收。遇到长任务，我会拆出检查点，不让模型一路跑到最后才发现方向已经偏了。

### 我对 Agent 的理解

在我看来，Agent 是一个能持续观察和执行的系统。它会根据目标选择工具，保存当前状态，再依据工具返回决定下一步。这个过程比普通对话更接近真实工作流。

但 Agent 并不会因为能调用工具就自动可靠。它可能看错返回结果，也可能在长任务里忘掉最初的限制。所以我会关注权限边界、任务拆分、运行日志、重试方式和人工确认。尤其是涉及发布、删除或对外发送内容时，我更愿意让它停下来问一句。

我期待的 Agent 不是永远自主，而是知道什么时候该继续，什么时候该把决定还给人。

### 知识库与 RAG

知识库最容易出现的问题，是文件放进去了，却没人认真整理。资料重复、版本过期或切分混乱，都会让检索结果变得不稳定。

我会先整理文档层级、来源和更新时间，再考虑分块、召回、重排与引用。回答能不能回到原文，对我来说很重要。否则模型说得再顺，也很难放心使用。

平时会用 NotebookLM 对照多份资料、做带引用的问答，用 Obsidian 保存长期笔记、项目记录和模板。需要持续使用的知识库，还要考虑版本管理、访问权限和定期抽查，而不是上传一次后就不再维护。
